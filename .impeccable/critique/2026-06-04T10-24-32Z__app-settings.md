## Anti-Patterns Verdict

**LLM assessment**: Clean. The settings page avoids every common AI slop marker. No side-stripe borders, no gradient text, no glassmorphism, no identical card grids, no eyebrow kickers, no numbered section markers. The restrained teal accent, single-family typography (Manrope), and flat-surface-first elevation strategy all read as authored, not generated. One subtle gradient exists on the BillingTab hero card (`bg-gradient-to-br from-teal/5 via-white to-teal/3`) but it's contextually justified on a financial summary surface.

**Deterministic scan**: The automated detector returned zero findings across all settings files. No structural anti-patterns detected.

## Overall Impression

Settings is one of the more composed surfaces in the app. The six-tab information architecture is well-ordered (business identity first, technical last), tabs hide correctly by role, and the visual language is consistent with the rest of AgencyViz. The biggest opportunity is **operational maturity**: the page handles setup well but doesn't help power users manage at scale (no search, no bulk ops, no keyboard shortcuts). The second opportunity is an accessibility gap: `text-faint` is used on informational descriptions that users need to read, but it fails WCAG AA contrast at 3.36:1.

## What's Working

1. **Information architecture scales with role.** Non-admin users see 4 tabs; owners see 6. The ordering (Company > Members > Roles > Billing > Integrations > Developer) mirrors how agencies think about their workspace: identity first, then team, then money, then technical plumbing. Nothing is hidden that shouldn't be; nothing is shown that creates confusion.

2. **High-stakes actions have appropriate friction.** Workspace deletion requires typing a confirmation phrase. API key generation shows a one-time copy warning ("This is the only time you'll see the full key"). Member removal triggers a confirmation dialog. These are the right guardrails at the right moments.

3. **Design system discipline is strong.** Teal appears on ~8-10% of interactive elements (CTAs, active tabs, toggles). Spacing uses the Tailwind scale consistently. Cards share the same border-radius and elevation vocabulary. The page feels cohesive without being monotonous.

## Priority Issues

### [P1] `text-faint` descriptions fail WCAG AA contrast

**What**: Section descriptions throughout Settings use `text-xs text-faint` or `text-sm text-faint`. The `text-faint` token (#8C8C8C) on white backgrounds produces a contrast ratio of 3.36:1, below the WCAG AA minimum of 4.5:1 for normal text.

**Where**: `SectionHeader` component (page.tsx:263), `DeveloperSection` descriptions (page.tsx:289), page subtitle (page.tsx:102-104), and propagated into MembersTab, BillingTab, and connector card descriptions.

**Why it matters**: These aren't decorative labels; they're informational descriptions that orient users ("Manage team access and per-member campaign notification defaults"). Users with moderate low vision or in bright ambient conditions will struggle to read them.

**Fix**: Promote descriptions from `text-faint` to `text-dim` (#6B7280, 4.83:1) or `text-prose` (#4B5563, 7.56:1). Reserve `text-faint` for truly optional content (timestamps, disabled labels, placeholder ghost text).

**Suggested command**: `/impeccable audit app/settings` (accessibility pass)

---

### [P2] Inline button violations erode component consistency

**What**: Approximately 30% of interactive CTA buttons use raw `<button>` elements with ad-hoc Tailwind classes instead of the `<Button>` primitive from `components/ui/Button.tsx`.

**Where**: ApiKeyManager (copy/generate buttons), WebhookEventCard (test/save buttons), BillingTab (portal management cards as clickable divs), GhlConnectorCard (multiple inline buttons), MetaConnectorCard (connect/disconnect buttons).

**Why it matters**: Inconsistent button rendering means inconsistent focus rings, hover states, disabled states, and loading spinners. The `<Button>` primitive handles all of these; inline buttons handle some or none. This is technical debt that compounds as the page grows.

**Fix**: Migrate all CTA-sized buttons to `<Button>` with appropriate variants (`primary`, `ghost`, `danger`, `outline`). Small icon-only actions (avatar camera overlay, inline copy icon) can stay as inline buttons with explicit `aria-label`.

**Suggested command**: `/impeccable polish app/settings`

---

### [P2] No search or progressive disclosure for scaling

**What**: MembersTab renders all team members in a flat list with no search input, no filtering, and notification toggles always expanded. The Developer tab's API key list has no timestamps or sort.

**Why it matters**: Agencies with 20+ team members will scroll past the entire list to find a specific person. Per-member notification configuration adds vertical density that compounds the problem.

**Fix**: Add a search/filter input at the top of MembersTab (filter by name or email). Collapse per-member notification toggles by default; expand on click. Add creation timestamps to API keys.

**Suggested command**: `/impeccable harden app/settings`

---

### [P2] No dirty-state indicators on forms

**What**: When a user modifies a form field (company name, business details, webhook URL), there's no visual signal that changes are unsaved. The save button appears/becomes active, but the form itself doesn't indicate "you have unsaved work."

**Why it matters**: Users who edit a field, switch tabs, then return won't know whether their changes were saved. This is especially risky for the Company profile tab where multiple fields can be edited simultaneously.

**Fix**: Add a subtle dirty-state indicator: either a dot on the tab label, a "Unsaved changes" banner below the section header, or a browser `beforeunload` warning when navigating away with dirty fields.

**Suggested command**: `/impeccable harden app/settings`

---

### [P3] Toggle switch implementations vary

**What**: GhlConnectorCard uses Lucide `ToggleRight`/`ToggleLeft` icons as toggle switches with proper `role="switch"` semantics. WebhookEventCard uses a custom CSS toggle that may lack `role="switch"` and `aria-checked`. MembersTab notification toggles use yet another pattern.

**Why it matters**: Three toggle implementations means three different keyboard behaviors, three different screen reader announcements, and three different visual treatments for the same interaction pattern.

**Fix**: Extract a shared `<Toggle>` component with consistent styling, `role="switch"`, `aria-checked`, and keyboard support (Space to toggle). Use it everywhere.

**Suggested command**: `/impeccable polish app/settings`

## Persona Red Flags

**Alex (Power User)**: No keyboard shortcuts for save (Enter) or cancel (Escape). Cannot bulk-change member roles. GHL mapping requires 10+ individual dropdown interactions with no auto-suggest or "apply defaults" shortcut. API key list shows no timestamps, so finding "the key I created last week" means reading every truncated key. Alex will build a spreadsheet to track what the UI should track.

**Sam (Accessibility-Dependent)**: `text-faint` descriptions at 3.36:1 contrast fail for low-vision users. Toggle switches lack consistent `role="switch"` and `aria-checked` semantics across components. Some icon-only buttons (avatar camera overlay, copy key) lack `aria-label`. The RolesTab permission matrix cells could use `aria-label` for screen readers ("Owner can manage team members: yes").

## Minor Observations

- Tab `description` fields are defined in the TABS array (page.tsx:34-39) but never rendered in the UI. Could serve as tooltips on the nav items.
- Role badge styling differs between MembersTab (`border-teal/20`) and RolesTab (`ring-1 ring-teal/20`). Should normalize.
- Save feedback varies: CompanyProfileCard shows a "Saved" checkmark for 2.5s; other forms only fade the loading spinner. Toast notifications would be more consistent.
- The `DeveloperSection` icon container uses `bg-wash` (page.tsx:283) which may be an undefined token if not in globals.css.
- Quote number format has a live preview in BusinessDetailsCard — excellent pattern that should be replicated for webhook URL validation.

## Questions to Consider

- What if the Members tab had a self-service mode where members could manage their own notification preferences, instead of requiring an admin to configure each person?
- The GHL connector wizard requires agencies to understand GHL pipeline structure before they start. What if the wizard fetched their actual GHL pipelines and pre-populated sensible defaults?
- Settings has no "export configuration" or "duplicate workspace setup." If an agency manages multiple workspaces (or rebuilds one), they reconfigure everything from scratch. Is that acceptable at the current user scale?
