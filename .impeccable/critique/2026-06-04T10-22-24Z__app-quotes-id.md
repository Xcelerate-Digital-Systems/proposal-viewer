## Anti-Patterns Verdict

**LLM assessment**: Clean. This does not read as AI-generated. No side-stripe borders, gradient text, glassmorphism, hero-metric cards, eyebrow abuse, or numbered section markers. The 13 SectionCards share a consistent shell (white rounded-2xl, border, header/body) but each has meaningfully different internal content (tables, forms, textareas, timelines, file uploaders). This is a design-system card primitive used correctly, not a stamped template.

## Overall Impression

The quote builder is a competent, well-engineered admin surface with a strong live preview and good domain-appropriate vocabulary. But it treats every section with equal weight, creating a 13-card scroll wall that punishes repeat users. The core financial sections (Line Items + Pricing) are buried at positions 6-7 of 13. Save behavior is inconsistent across sections, and there is no undo despite the provider being wired up. The biggest opportunity: section grouping and navigation to turn a long form into a structured workspace.

## What's Working

1. **Live preview is excellent.** The sticky PreviewPane renders the actual QuoteSinglePageView with real data, fonts, and branding. Changes reflect immediately. This removes the "what will the customer see?" question entirely and is the builder's best feature.

2. **The saved-customer dropdown is thoughtful.** It queries prior proposals, deduplicates by name+email+phone, and auto-fills all 5 fields on selection. The Real/Test mode toggle with snapshot/restore is a genuinely useful workflow for agencies testing quote designs.

3. **LineItemsLibraryBar provides meaningful productivity.** Template save/load for line items helps agencies who build similar quotes repeatedly. The save flow (name, confirmation) and load flow (dropdown with delete-on-hover) are compact and well-placed in the SectionCard header.

## Priority Issues

### [P1] 13-section scroll wall with no grouping or navigation

The 13 sections in a flat `space-y-5` stack exceed cognitive chunking limits (3/8 on cognitive load checklist). Users have no way to jump to Line Items (the most-used section) without scrolling past 5 other sections. Only Activity collapses.

**Why it matters**: Agency owners building 10 quotes/day feel the scroll tax on every quote. The builder optimizes for first-time discovery at the expense of repeat efficiency.

**Fix**: Group sections into 3-4 collapsible clusters: "Client & Project" (Activity, Client, Project Details, Scope, Photos), "Financials" (Line Items, Pricing), "Branding & Trust" (About Us, Testimonial, Badges), "Closing" (Next Steps, Terms, Attachments). Add sticky section nav or anchor links. Consider collapsing non-Financial groups by default.

### [P1] Save behavior inconsistency across sections

Scope auto-saves on blur AND has a Save button. Pricing toggles auto-save but deposit % needs blur + save button. Project Details requires explicit save. Client Details requires explicit save. This creates user uncertainty about whether work is persisted.

**Why it matters**: Users cannot build a reliable mental model of "how saving works." The EditorSaveStatusBadge in the header helps but is too far from the point of edit.

**Fix**: Unify to auto-save on debounced blur for all fields. Remove explicit Save buttons. Show inline "Saved" indicators near each section. Extend the useReportSaveStatus pattern to all sections.

### [P2] Status change to Draft silently destroys timestamps

Changing status from Sent/Accepted/Declined back to Draft nulls `first_viewed_at`, `last_viewed_at`, and `sent_at` with no confirmation dialog and no undo.

**Why it matters**: Accidentally selecting "Draft" destroys analytics data. The user gets a success toast with no indication that data was lost.

**Fix**: Add a confirmation dialog when transitioning to Draft that warns about data loss. Use the existing `confirm()` pattern from the delete flow.

**Files**: [QuoteShellHeader.tsx:83-87](components/admin/quotes/QuoteShellHeader.tsx#L83-L87)

**Suggested command**: `/impeccable harden app/quotes/[id]`

### [P2] No keyboard shortcuts or accelerators

No Cmd+S, no Cmd+Enter, no section jump shortcuts, no Tab-to-next-row in the line items table. The power user path is scroll + click + type + scroll.

**Why it matters**: The Alex persona building 10 quotes/day is bottlenecked by mouse-only interactions.

**Fix**: Add Cmd+S to force-save all dirty sections. Make line items table Tab-navigable in row order (skip Total and Delete columns). Add Enter on last row to auto-add a new row.

### [P3] Three header buttons bypass the Button primitive

**Why it matters**: Inconsistency in the most prominent UI element. Missing keyboard focus indicators for accessibility.

**Fix**: Convert to `<Button variant="ghost" size="sm" iconOnly leftIcon={...} aria-label="..." />`.

## Persona Red Flags

**Alex (Power User -- 10 quotes/day)**: No keyboard shortcuts anywhere. Cannot duplicate or reorder line items. Cannot jump to sections. Scroll-heavy flow with no accelerators. The saved-customer dropdown and line items library are genuine time-savers, but the surrounding workflow penalizes speed. High friction for repeat use.

**Sam (Accessibility-Dependent)**: Focus indicators are subtle (`focus:ring-1 focus:ring-teal/30` at 30% opacity on white). Three icon-only header buttons have `title` but no `aria-label`. Line items table uses CSS Grid divs instead of `<table>` semantics -- screen readers won't announce it as a data table. Activity collapse toggle has no `aria-expanded` attribute.

**Riley (Stress Tester)**: 50 line items will lag -- `updateItem` maps over ALL items on every keystroke with no virtualization. PricingSettingsSection fetches line items independently from QuoteLineItemsSection, causing stale subtotals until next refetch. Rapid GST toggle fires concurrent saves with no debounce. Deposit % accepts 0 and >100 with no validation.

## Minor Observations

- `PricingSettingsSection.tsx:191-199`: inline `<button>` should use `<Button variant="primary">`.
- `QuoteLineItemsSection.tsx:116`: `divide-gray-50` uses raw Tailwind gray instead of semantic `divide-edge` token.
- PricingSettingsSection fetches line items independently from QuoteLineItemsSection -- editing a line item won't update the Pricing subtotal until the next refetch cycle. Stale financial data in a financial tool.
- `CurrencyInput` on unit price but raw `<input type="number">` on quantity creates a visual inconsistency for large numbers.
- `LineItemsLibraryBar` dropdown uses `absolute z-50` positioning -- could be clipped by scroll containers. Other dropdowns use `createPortal`.

## Questions to Consider

1. Why is this a vertical scroll of 13 sections rather than a stepped or sub-tabbed flow? The tab infrastructure already exists. The builder tab alone could split into "Content" and "Pricing" sub-views, cutting scroll depth in half.

2. The Pricing section is statefully separated from Line Items, but they are a single financial unit. When the subtotal shown in Pricing is stale because the user just edited a line item two sections above, does that erode trust in the tool's accuracy?

3. Why does EditorUndoProvider exist in the layout but zero sections consume it? If undo is important enough to have a provider, it is important enough to use.
